# Sentinel GuardTeam — 支付风控多智能体自主闭环

GOAI · 新智基座 / Agent Infra 赛道 · 参考方向 #4「金融风控与理赔自动化」

> 一个基于 **AgentTeams** 的多 Agent 风控闭环,由自研运行时安全执行层 **Sentinel**
> 提供"安全 + 合规审计"骨干 Skill。核心主张:**不只是能做风控的多 Agent 系统,更是
> Agent 自己不会被提示注入劫持去执行越权/欺诈支付的系统。**

---

## 1. 场景与痛点(对应 25% 场景价值)

金融风控与理赔自动化正在被多 Agent 系统接管:交易监控、反欺诈、理赔核验。但一旦 Agent
能"动钱",一个新风险出现——**被投毒的工单/页面/上游数据,可以让风控 Agent 自己把款项
放给攻击者**(即 Aim Security 命名的 *LLM 作用域越权*,EchoLeak / CVE-2025-32711 一类)。
现有多 Agent 框架解决"怎么协同",不解决"协同过程中的动作是否可信"。

**GuardTeam 让风控闭环既自动、又可信、且可审计。** 可迁移到反欺诈、理赔核验、支付合规。

## 2. 多 Agent 自主闭环(对应 25% 多 Agent 协同)

基于 AgentTeams 的 **Manager-Workers** 架构,由 Manager 编排 **4 个不同职能 Agent**,
形成 参考方向 #4 要求的完整闭环:

```
        ┌─────────────── AgentTeams Manager(编排 / 人在回路)───────────────┐
信号 →  │  ① 信号聚合 Agent  →  ② 风险定位 Agent  →  ③ 处置方案 Agent  →  ④ 合规审计 Agent │ → 结论
        └──────────────  Sentinel Skills(每个高危动作结算前强制校验)  ─────────────┘
```

| Agent(Worker) | 职能 | 关键能力 |
|---|---|---|
| ① 信号聚合 | 汇聚交易流水、告警、工单、上游数据 | 多渠道归集、去重、结构化 |
| ② 风险定位 | 定位可疑交易与根因 | 结合 **provenance 来源追踪**,判断可疑参数是否来自不可信内容 |
| ③ 处置方案 | 生成 拦截/放行/转人工 的处置动作 | 调用 **Sentinel mandate 校验**,越权即熔断 |
| ④ 合规审计 | 对每一步产出防篡改证据 | 生成 **签名回执**,形成可审计闭环 |

异常处理:任一 Agent 的高危动作(放款/转账/授权)在执行前经 Sentinel 校验,越权、超预算、
或参数溯源到污点内容 → **结算前熔断 + 留痕**,并可 `@人工` 介入(AgentTeams 人在回路)。

## 3. Skill 工程体系(对应 25% Skill 工程 + 生态复用)

Sentinel 的能力封装为一组**可复用 Skill**,通过 **MCP** 接入每个 Worker(AgentTeams 原生
支持声明式 MCP)。每个 Skill 都有清晰的输入/输出、调用条件、依赖与失败处理:

| Skill | 输入 → 输出 | 调用条件 | 失败处理 | 复用价值 |
|---|---|---|---|---|
| `mandate.check` | (交易, 签名授权) → 允许/拒绝 + 理由 | 任一高危动作前 | 签名不符 → 拒绝 | 任何"agent 花钱"场景 |
| `provenance.trace` | (动作参数, 会话) → 是否溯源到污点 | 处置/放款前 | 无污点源 → 放行 | 反注入、反欺诈 |
| `receipt.sign` | (决策) → HMAC 签名回执 | 每次决策后 | — | 合规审计、纠纷取证 |
| `injection.scan` | (不可信文本) → 注入判定(中英文) | 摄入外部数据时 | — | 通用注入防护 |

**这四个 Skill 已经实现、可运行、有 46 项测试与两套基准(SentinelBench / CommerceBench)**——
不是设想,是现成资产(见 `github.com/jiangmuran/sentinel`)。

## 4. 工程落地、运行验证与安全可审计(对应 20%,我们的强项)

- **可运行 Demo**:`python examples/commerce_demo.py` 现场演示"投毒 → 越权支付被熔断 / 合法通过"。
- **日志 / Trace 证据**:每个决策写入结构化 JSONL 审计日志(含污点 id、匹配 token、理由)。
- **权限审计完整**:签名 mandate(额度/白名单/有效期/累计预算)+ 防篡改签名回执。
- **真机验证**:已在官方 MCP SDK 上端到端跑通(真 `mcp.ClientSession` → 代理 → 真 server)。

## 6. 与赛道要求的对齐(自检)

| 硬性要求 | 我们的满足方式 |
|---|---|
| ≥3 个不同职能 Agent 的完整闭环 | 4 个 Agent:聚合→定位→处置→审计 |
| 必须用 AgentTeams 作为协同基点 | Manager-Workers 编排 + Matrix 人在回路 + MinIO 共享上下文 |
| Skill 为必选项 | 4 个 Sentinel Skills(输入/输出/失败处理/复用价值齐备) |
| 推荐:MCP 集成 / 可观测 | MCP 原生接入;JSONL 审计日志 = Trace 证据 |

## 7. 差异化(为什么评委记得住)

**"这是唯一一个 Agent 自己不会被劫持去乱付款的风控系统。"** 别的队做"能协同的多 Agent",
我们做"协同过程中每个高危动作都被运行时强制校验授权与来源"的多 Agent——直接吃满
20% 的"安全可审计",而且是别人没有的护城河(Sentinel 已开源、已验证)。

**【项目名称】** Sentinel GuardTeam — 支付/理赔风控多智能体自主闭环

**【问题场景】** 风控理赔已由多 Agent 接管;Agent 能动钱后,一张投毒工单即可操纵它越权放款(LLM 作用域越权)。现有框架只管"协同",不管"动作可信"。

**【核心方案】** 基于 AgentTeams 编排四个 Agent 成闭环:①信号聚合(注入检测+打污点)→②风险定位(来源追踪+黑名单/频次/金额筛查)→③处置(放款/复核/拒绝)→④合规审计(结算前强制校验+签名回执)。三态:通过/拦截/暂缓复核。

**【创新】**「LLM 提议,Sentinel 裁决」:分析 Agent 可 LLM 驱动,但每笔放款强制确定性执行——越权/超预算/参数溯源到污点即熔断。是唯一自身不会被注入劫持乱付款的风控系统,直吃"安全可审计"。

**【开放复用】** 四个 Skill(注入检测/溯源/授权校验/签名回执)经 MCP 接入、可跨 Team 复用;Apache-2.0,SentinelBench+CommerceBench 开放可复现。

**【进展】** 闭环、四 Skill、CLI、防篡改审计账本、合规报告已实现;80 测试全绿,真 MCP 客户端验证,已容器化;复赛部署 AgentTeams。
